# Specification: `category-suggester` Plugin

## Overview

A Next.js plugin for the `aj` microkernel platform that adds a **product detail tab** to every product page. On load, it calls an LLM (via LiteLLM + BAML) to evaluate whether the product's assigned category is correct and renders the verdict with a confidence badge. There is no caching — the LLM is called fresh on every tab open.

### 8.3 `baml_src/main.baml`

```
class CategoryInfo {
  name string
  description string?
}

class CategorySuggestion {
  isCorrect bool
  confidence float
  reasoning string
  suggestedCategoryName string?
}

function SuggestCategoryCorrectness(
  productName: string,
  productDescription: string?,
  currentCategoryName: string,
  currentCategoryDescription: string?,
  availableCategories: CategoryInfo[]
) -> CategorySuggestion {
  client LlmProvider
  prompt #"
    {{ _.role("system") }}
    You are a product categorisation expert. Your job is to decide whether
    a product belongs in its currently assigned category, and — if not — to
    suggest the best alternative from the provided list.

    Rules:
    - Base your judgement solely on the product name, its description, and the
      category names/descriptions you are given.
    - "confidence" must be a float between 0.0 and 1.0 that reflects how sure
      you are that your verdict is correct.
    - "isCorrect" is true when the current category is the best fit.
    - "suggestedCategoryName" must be null when isCorrect is true, and must
      exactly match one of the names in availableCategories when isCorrect is
      false.
    - "reasoning" must be a concise explanation (1–3 sentences) of your verdict.

    {{ _.role("user") }}
    Product name: {{ productName }}
    {% if productDescription %}
    Product description: {{ productDescription }}
    {% endif %}

    Current category: {{ currentCategoryName }}
    {% if currentCategoryDescription %}
    Current category description: {{ currentCategoryDescription }}
    {% endif %}

    Available categories:
    {% for cat in availableCategories %}
    - {{ cat.name }}{% if cat.description %}: {{ cat.description }}{% endif %}

    {% endfor %}

    {{ ctx.output_format }}
  "#
}
```

## 9. API Route — `src/pages/api/suggest-category.ts`

### Purpose

Server-side Next.js API route. Fetches the product and all categories from the host app, calls the BAML function, and returns the structured suggestion. **No result is persisted** — this is a pure read-and-infer operation.

### Method

`POST` only. Returns `405` for any other method.

### Request body

```typescript
{
  productId: string | number   // required
}
```

Validation: `productId` must be present and non-empty after `String(...).trim()`. Return `400` with `{ error: "Missing required fields", details: ["productId"] }` if absent.

### Response body (success — `200`)

```typescript
{
  isCorrect: boolean
  confidence: number          // 0.0 – 1.0
  reasoning: string
  suggestedCategoryName: string | null
}
```

### Error responses

| Status | Condition | Body |
|--------|-----------|------|
| `400` | Missing `productId` | `{ error: "Missing required fields", details: ["productId"] }` |
| `405` | Non-POST method | `{ error: "Method not allowed" }` |
| `4xx` | Host API error (product/categories fetch failed) | `{ error: "<host error message>" }` — status code forwarded from the host |
| `500` | LLM call failed or unexpected error | `{ error: "<error message>" }` |

### Implementation steps

1. Parse `productId` from `req.body`.
2. Create the server SDK: `createServerSDK("category-suggester", undefined, req)`.
3. Fetch the product via `sdk.hostApp.getProduct(productId)`.
   - The response shape is `{ id, name, description, categoryId, categoryName, ... }`.
   - If the product has no category assigned (`categoryId` is null/undefined), return `400`: `{ error: "Product has no category assigned" }`.
4. Fetch all categories via `sdk.hostApp.fetch("/api/categories")` — returns `Response`; call `.json()` to get an array of `{ id, name, description, ... }`.
5. Find the current category object in the categories array by matching `cat.id === product.categoryId` (or `cat.name === product.categoryName` as fallback).
   - If not found, return `400`: `{ error: "Current category not found in category list" }`.
6. Build `availableCategories: CategoryInfo[]` from the full categories array — include every category's `name` and `description` (omit `id`).
7. Call `b.SuggestCategoryCorrectness(...)` with the assembled arguments.
8. Return `200` with `{ isCorrect, confidence, reasoning, suggestedCategoryName }`.
9. Wrap steps 2–8 in a try/catch. On error, extract the status code from `"Host API error <N>"` pattern (same pattern as `ai-description`); default to `500`.

### Full TypeScript interface declarations (for the implementer)

```typescript
interface SuggestRequest {
  productId: string | number;
}

interface ErrorResponse {
  error: string;
  details?: string[];
}

interface SuggestResponse {
  isCorrect: boolean;
  confidence: number;
  reasoning: string;
  suggestedCategoryName: string | null;
}

// Minimal shapes expected from the host API
interface HostProduct {
  id: number;
  name: string;
  description?: string;
  categoryId?: number;
  categoryName?: string;
}

interface HostCategory {
  id: number;
  name: string;
  description?: string;
}
```

## 10. React Component — `src/pages/product-tab.tsx`

### Behaviour

- On mount, **immediately and automatically** calls `/api/suggest-category` with `productId` — no button press required.
- No caching: every mount triggers a fresh LLM call.
- There is no "generate" or "regenerate" button — the check runs automatically.

### State

```typescript
const [result, setResult] = useState<SuggestResult | null>(null);  // from domain.ts
const [loading, setLoading] = useState(true);
const [error, setError] = useState<string | null>(null);
```

### Loading state

While `loading === true`, render:

```tsx
<div className="tc-plugin" style={{ padding: "1rem" }}>Loading...</div>
```

### Error state

If `error` is non-null (and not loading), render inside the main layout:

```tsx
<p className="tc-error">{error}</p>
```

### Result display

The result card renders inside a `tc-card` and shows:

1. **Verdict line** — a single sentence:
   - If `isCorrect`: `"✓ Category looks correct"`
   - If not `isCorrect` and `suggestedCategoryName` is present: `"✗ Category may be incorrect — consider: {suggestedCategoryName}"`
   - If not `isCorrect` and no suggestion: `"✗ Category may be incorrect"`

2. **Confidence badge** — uses the host's badge CSS classes:
   - `confidence >= 0.7` → `tc-badge tc-badge--success` — label `"{(confidence * 100).toFixed(0)}% confidence"`
   - `0.5 <= confidence < 0.7` → `tc-badge tc-badge--warning` — same label format
   - `confidence < 0.5` → `tc-badge tc-badge--danger` — same label format

   > **Note:** The host CSS currently defines `--success` (green) and `--danger` (red). Inspect `plugin-ui.css` to confirm whether `--warning` (amber) exists. If it does not, implement the amber badge with an inline `style` override: `style={{ backgroundColor: "#fef3c7", color: "#92400e", ...existingBadgeStyles }}`. The implementer must check at development time.

3. **Reasoning** — a paragraph of plain text below the badge.

### Full layout sketch

```tsx
<div className="tc-plugin" style={{ padding: "1.5rem" }}>
  <h3 style={{ margin: "0 0 1rem" }}>Category Check</h3>

  {error && <p className="tc-error">{error}</p>}

  {result && (
    <div className="tc-card" style={{ padding: "1rem" }}>
      <p style={{ margin: "0 0 0.5rem", fontWeight: 500 }}>
        {verdictText}
      </p>
      <span className={badgeClass} style={badgeInlineStyle}>
        {confidenceLabel}
      </span>
      <p style={{ margin: "0.75rem 0 0", color: "#555" }}>
        {result.reasoning}
      </p>
    </div>
  )}
</div>
```

### `useEffect` logic

```typescript
useEffect(() => {
  if (!sdk) return;
  if (!productId) {
    setError("Product ID is missing. This tab must be opened from a product detail page.");
    setLoading(false);
    return;
  }

  async function run() {
    try {
      const token = await sdk!.hostApp.getToken();
      const response = await fetch("/api/suggest-category", {
        method: "POST",
        headers: {
          "Content-Type": "application/json",
          ...(token ? { Authorization: `Bearer ${token}` } : {}),
        },
        body: JSON.stringify({ productId }),
      });

      if (!response.ok) {
        const body = (await response.json()) as { error?: string };
        throw new Error(body.error ?? "Failed to check category.");
      }

      const data = (await response.json()) as SuggestResponse;
      setResult(toSuggestResult(data));
    } catch (err) {
      if (err instanceof TypeError && err.message === "Failed to fetch") {
        setError("Network error. Please check your connection and try again.");
      } else {
        setError(err instanceof Error ? err.message : "An unexpected error occurred.");
      }
    } finally {
      setLoading(false);
    }
  }

  void run();
}, [productId, sdk]);
```

> The dependency array is `[productId, sdk]`. Since the component mounts once per tab open, this achieves "always re-evaluate on load".

## 11. `src/domain.ts`

Holds the plugin-specific result type and a mapper. There is no `PluginObject` mapping needed (results are never persisted).

```typescript
export interface SuggestResult {
  isCorrect: boolean;
  confidence: number;
  reasoning: string;
  suggestedCategoryName: string | null;
}

export function toSuggestResult(raw: {
  isCorrect: boolean;
  confidence: number;
  reasoning: string;
  suggestedCategoryName?: string | null;
}): SuggestResult {
  return {
    isCorrect: raw.isCorrect,
    confidence: raw.confidence,
    reasoning: raw.reasoning,
    suggestedCategoryName: raw.suggestedCategoryName ?? null,
  };
}
```

## 13. Testing Approach (Manual Verification)

### 13.1 Setup

1. Start the host: `./mvnw spring-boot:run` (from the repo root).
2. Ensure LiteLLM proxy is running on `http://localhost:4000` and `LITELLM_API_KEY` is set in `.env`.
3. Install dependencies and generate the BAML client: `cd plugins/category-suggester && npm install` (triggers `postinstall`).
4. Start the dev server: `npm run dev`.
5. Register the manifest: `curl -X PUT http://localhost:8080/api/plugins/category-suggester/manifest -H "Content-Type: application/json" -d @manifest.json`.

### 13.2 Happy-path checks

| # | Steps | Expected |
|---|-------|----------|
| 1 | Open any product that has a category assigned. Click the "Category Check" tab. | "Loading..." appears briefly, then the result card renders. |
| 2 | Result card is visible. | Verdict text, confidence badge, and reasoning paragraph all present. |
| 3 | Confidence ≥ 0.7. | Badge is green (`tc-badge--success`). |
| 4 | Confidence between 0.5 and 0.69. | Badge is amber. |
| 5 | Confidence < 0.5. | Badge is red (`tc-badge--danger`). |
| 6 | `isCorrect` is `false`. | Verdict text includes `"consider: <name>"` when `suggestedCategoryName` is provided. |
| 7 | Navigate away from the tab and back. | LLM is called again (no cached result shown). |

### 13.3 Edge-case checks

| # | Steps | Expected |
|---|-------|----------|
| 8 | Open a product with **no category** assigned. | Error message: `"Product has no category assigned"`. |
| 9 | Stop LiteLLM proxy; open tab. | Error message from the caught exception is displayed via `tc-error`. |
| 10 | Directly `POST /api/suggest-category` with no body. | `400` with `{ error: "Missing required fields", details: ["productId"] }`. |
| 11 | `POST /api/suggest-category` with a non-existent `productId`. | `404` (forwarded from host). |
| 12 | `GET /api/suggest-category`. | `405 Method not allowed`. |

## 14. Implementation Notes & Decisions

- **No caching by design.** Category assignments change, and LLM calls are cheap via the proxy. The spec intentionally omits any `thisPlugin.objects` persistence.
- **`sdk.hostApp.fetch` vs raw `fetch` in API route.** The API route runs server-side in Node.js (not in an iframe), so it uses `createServerSDK` and calls `sdk.hostApp.getProduct` / `sdk.hostApp.fetch` to talk to the host over HTTP with the forwarded auth token. Never call the host from the browser-side component directly.
- **Auth token forwarding.** The frontend reads the token via `sdk.hostApp.getToken()` and passes it in the `Authorization` header of the `POST` to `/api/suggest-category`. The server SDK then forwards it to the host API calls.
- **`suggestedCategoryName` nullability.** The BAML function returns `string?` which becomes `string | null` in TypeScript. Always default to `null` in the mapper.
- **Port selection.** `3005` is used. If another plugin already occupies this port, update both `package.json` (`dev`/`start` scripts) and `manifest.json` (`url` field) consistently.
